# 당도(브릭스) 등급 세분화 — 설계 문서

> 목적: 현재 '일반/고당' 2등급을 '일반 + 사용자 정의 브릭스 등급 여러 개(예: 11브↑/12브↑/13브↑)'로 확장. 지금의 우회법(농가명에 "11.5브" 위장 + 고당 지웠다 재입력)을 정식 기능으로 대체하고, 재고현황에서 등급별로 한눈에(중량 자동 환산 포함) 표시.
>
> 성격: **큰 작업.** quality_grade가 약 68곳에 '고당'/'일반' 2값 전제로 하드코딩됨. 운영 중이므로 단계별로 잘게, 각 단계 화면+SQL 검증. 베타라 기존 데이터 마이그레이션은 자유(불일치 허용).

---

## 1. 현재 상태 (사실)

- `quality_grade`: '일반' / '고당' 2값. inventory_records·sorting_details에 저장. 약 68곳에서 하드코딩 비교('고당'/'일반').
- 등급 탭(`_invGrade`): 'all' / '고당' / '일반' — 재고현황에서 이미 등급 필터 존재.
- `kgPerCt(품목)`: 품목별 중량 헬퍼 존재(productWeights). 전체현황에서 이미 CT→kg 환산 사용 → 재고현황 중량 자동표기 **가능**.
- 부분적 브릭스 개념 이미 존재: 품질 기준 설정에 `brix_high_min`/`brix_mid_min`(당도 상/중 최소값), 레코드에 `brix_grade`/`brix_range` 필드. (단, 이건 개별 품질 표기용이고 재고 등급 분리와는 별개.)
- 현재 우회법: 선과처리로 일반/고당 등록 → 고당분 삭제 → 농가명을 "11.5브릭스" 등으로 골라 재입력. 농가명 오염·데이터 이중 이동·수기 엑셀 관리 유발.

---

## 2. 목표 설계

### 2-1. 데이터 모델
- **브릭스 등급 마스터 테이블 신설** (예: `brix_grades`): { id, label(예: "12브릭스 이상"), sort_order, is_active }. 설정에서 관리.
  - '일반'은 등급이 아니라 기본값(등급 미지정)으로 유지 → 기존 로직 호환.
  - 등급은 sort_order로 정렬(낮은 당도 → 높은 당도).
- **quality_grade 값 확장**: '일반' | (브릭스 등급 label). 즉 기존 '고당' 자리에 사용자 정의 등급명이 들어감.
  - 판별을 "=== '고당'"에서 "!== '일반'(=등급 있음)"으로 일반화. ★이게 68곳 수정의 핵심 방향.
- 기존 '고당' 데이터: 베타라 자유. 마이그레이션 옵션 (가) 그대로 '고당' 등급으로 두기 (나) 일괄 특정 브릭스 등급으로 변환 (다) 삭제. → 구현 시 결정.

### 2-2. 브릭스 경계값 기록
- 경계는 유동적(9/11/12/13 등) → 등급 label 자체에 "11브릭스 이상"처럼 기록(숫자 하드코딩 안 함).
- 선과 배치별로 다른 경계를 썼다면 label로 구분(같은 "12브릭스 이상"이면 동일 등급으로 합산).
- (선택) 배치에 실제 측정 브릭스값 메모 필드 추가는 2차.

### 2-3. 화면 3곳
1. **설정 — 당도 등급 관리**: 등급 목록 CRUD(추가/삭제/순서). 기존 품질 기준 설정 근처.
2. **선과 처리 — 등급 선택**: 고당 체크/버튼 대신 '당도 등급' 드롭다운(일반 + 등급들). sizeDetails에 quality_grade=선택 등급 저장. 사이즈별 CT 입력은 기존과 동일.
3. **재고현황 — 등급별 표시**:
   - 등급 탭: 전체 / 일반 / 등급들(동적). 기존 `_invGrade` 확장.
   - 등급별로 농가·사이즈 매트릭스(기존 renderInventoryStatus 재사용).
   - **중량 자동표기**: 각 셀·합계에 CT×kgPerCt(품목) 병기(엑셀의 "CT당 약 18KG → kg 합계"). kgPerCt 재사용.

### 2-4. 출고 (2차, 지금은 보류)
- 출고 시 등급 구분은 유용하나 구현 부담 큼. 1차는 재고 등록·표시까지. 출고는 quality_grade가 이미 출고 레코드에 흐르므로 2차에 표시만 확장.

## 3. 단계별 구현 순서 (운영 안 깨지게 잘게)

**0단계 — 판별 일반화 (선행 리팩토링, 위험 관리 핵심)**
- 68곳의 `quality_grade === '고당'` / `=== '일반'` 을 헬퍼로 통일: `isNormalGrade(r)` = (quality_grade||'일반')==='일반', `gradeOf(r)` = quality_grade||'일반'.
- 동작 100% 동일하게(고당/일반 2값 그대로) 리팩토링만. 이후 단계에서 값만 늘리면 되게.
- ★이 단계가 안전판. 검증: 기존 고당/일반 화면 전부 그대로.

**1단계 — 브릭스 등급 마스터 + 설정 화면**
- brix_grades 테이블 생성. 설정에 당도 등급 관리 UI(추가/삭제/순서).
- 데이터만, 아직 선과/재고에 안 씀. 위험 낮음.

**2단계 — 선과 처리 등급 선택**
- 고당 대신 등급 드롭다운. sizeDetails.quality_grade에 등급 label 저장.
- 검증: 등급 선택 선과 → sorting_details/inventory_records에 label 저장 확인(SQL).

**3단계 — 재고현황 등급 탭 + 표시**
- `_invGrade`를 동적 등급으로 확장. 등급별 매트릭스.
- 검증: 등급별 재고가 맞게 분리·합산되는지(SQL 대조).

**4단계 — 중량 자동표기**
- 재고현황 매트릭스에 CT×kgPerCt 병기. 합계 kg.
- 검증: 몇 개 셀 수기 계산과 일치.

**5단계 — 기존 우회 데이터 정리 (선택)**
- 농가명 "11.5브" 위장 데이터를 정식 등급으로 이관 또는 정리. 베타라 자유.

**(2차) 출고 등급 표시**

## 4. 위험 지점

- **quality_grade 68곳**: 0단계(판별 일반화)를 반드시 먼저. 안 하면 값 늘릴 때 곳곳에서 '고당' 조건이 등급을 놓침.
- **등급 분리 = 과거 데이터 손실 버그 영역**: 한 등급 작업이 다른 등급 건드리면 손실. 단계마다 SQL로 등급별 합계 대조.
- **삭제·편집 모달의 등급 분기**(4883 "고당만 삭제" 등): 다등급이면 "고당만" → "이 등급만"으로 일반화 필요.
- **선과 결과 모달**(고당/일반 블록): 다등급이면 블록도 등급 수만큼. 12933/12963 normalMapH 등 확장.
- **운영 중**: 각 단계 후 화면+SQL 검증, git tag 백업.
- **브릭스 부분 개념(brix_high_min 등)과 혼동 주의**: 그건 개별 품질 표기용. 새 등급 분리와 별개로 둘지 통합할지 1단계에서 결정.

---

## 5. 확정된 결정

- **기존 '고당' 데이터**: 특정 브릭스 등급으로 **일괄 변환**(어느 등급으로는 구현 때, 예: "11브릭스 이상"). 베타라 부담 없음.
- **'일반'**: 등급이 아니라 **기본값(미지정)**으로 유지 → 기존 로직 호환.
- **재고현황 표시**: **세로 나열**(일반 → 등급들 순으로 표를 세로로 쭉, 엑셀 스크린샷처럼 한눈에) + 상단 등급 탭은 필터 겸용(특정 등급만 보고 싶을 때). 기본은 전체 세로 나열. 인쇄(보고서)에도 세로 나열이 적합.
- **부분 브릭스 개념(brix_grade/brix_high_min 등)**: **별개 유지, 통합 안 함.** 용도가 다름 — 기존 brix_grade/brix_range는 입고분의 품질 꼬리표(당상/당중/당하, 참고 정보)이고, 새 브릭스 등급은 재고를 등급별로 나눠 세는 재고 분리(quality_grade 확장). 통합하면 오히려 꼬임.

## 6. 남은 확인 (구현 착수 시)

- 기존 '고당'을 어느 브릭스 등급으로 변환할지(기본 후보: 첫 브릭스 등급).
- 재고현황 세로 나열 시 등급 순서(sort_order: 일반 먼저 or 마지막).

---

## 7. ★ 1차 범위 확정 — 재고현황 중심 (앞단은 나중에)

선과 입력·비율·저당 분리 등 앞단은 손볼 곳이 너무 많고 위험 → **1차는 재고현황만**. quality_grade를 등급명으로 확장해두면, 나중에 선과 입력이 이 필드를 자동으로 채우게 연결만 하면 앞단까지 완성(확장성 확보). 앞단(선과 한 번에 일반+고당 동시 산출, 저당이 일반에 섞임 등)은 실제 작업 흐름 재확인 후 2차 설계.

### 1차에 하는 것
1. 브릭스 등급 설정(정의).
2. 재고 편집에서 등급 **수동 지정**(앞단 없이 지금 바로 정식 재고 입력).
3. 재고현황 **등급별 표시**: 등급마다 표 하나씩 세로로. 표 구조 = 가로 선과일자(농가) · 세로 사이즈 · 오른쪽 사이즈별 합계 CT + 중량 kg(합계 CT×kgPerCt) · 맨 아래 전체 합계.
   - CT/kg 배치(나란히 vs 위아래)는 구현 때 데모 A/B로 결정 — 데이터는 동일(사이즈별 합계 CT + kg 환산).

### quality_grade 판별 패턴 (0단계 대상, 실측)
- (a) `=== '고당'` 13곳 → `gradeOf(r) !== '일반'`(등급 있음).
- (b) `=== '일반'` 5곳 → `isNormal(r)`.
- (c) `|| '일반'` 폴백 19곳 → 그대로 유지 가능(등급값 없으면 일반).
- (d) `['일반','고당']` 순회 3곳(5660/5670/7272) → 동적 등급 목록(일반 + 활성 브릭스 등급)으로.

### 1차 단계 순서 (진행 상태)
- **0단계 ✅완료(v302)**: 판별 일반화 — gradeOf/isNormalGrade/isGraded 헬퍼 도입(app.js 834~836), 레코드 판별 12곳 치환. 동작 100% 동일. _invGrade UI탭·['일반','고당'] 순회(5660/5670/7272)·저장은 안 건드림(다음 단계). git tag brix-step0-before.
- **1단계 ✅완료(v303)**: brix_grades 마스터(테이블+RLS/anon권한, pachi_usages와 동일) + db.js 4함수(dbGetBrixGrades 등) + 전역 brixGrades + 설정 UI(당도 등급 관리 카드, renderBrixGradeCfg/add/edit/delete/toggle). 설정→파치 사용처 화면 하단 배치. 빈 값이면 표시 안 함. 등급 추가→DB 저장 검증 완료.
- **2단계 (다음)**: 재고 편집 모달에 등급 드롭다운(일반 + brixGrades) 수동 지정. 저장 시 quality_grade=등급 label. 이때부터 재고에 브릭스 등급 실제 입력.
- **3단계**: 재고현황 등급별 표(세로 나열, 일자·사이즈, 합계 CT). _invGrade 동적 확장.
- **4단계**: 중량 kg 자동표기(CT×kgPerCt).
- **5단계(선택)**: 기존 '고당'·농가명 위장 데이터 정리.

각 단계 화면+SQL 검증, git tag 백업. 여러 세션에 걸쳐 진행.
